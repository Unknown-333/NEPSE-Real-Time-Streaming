## 1. What This Project Is

**One-liner:** A real-time streaming data pipeline that takes stock market trade data, processes it into candlestick charts (OHLCV), stores it in a time-series database, and visualizes it on a live dashboard — all end-to-end.

**In plain English:**
Imagine you're watching a stock market. Every second, hundreds of trades happen — "Person A bought 100 shares of NABIL at Rs. 1150." This project takes all those individual trades (12.5 million of them), groups them into 1-minute summaries ("In minute 11:00-11:01, NABIL opened at 1150, went up to 1155, dropped to 1148, and closed at 1152 with 5000 shares traded"), stores them in a database optimized for time-series data, and shows them as live candlestick charts on a dashboard.

**Why it matters:** This is exactly how real trading platforms (like Binance, TradingView, Bloomberg) work behind the scenes. Building this demonstrates you understand:
- Real-time data streaming
- Stream processing with windowed aggregations
- Time-series databases
- Data pipeline architecture
- Dashboard visualization

## 2. The Problem It Solves

**Business problem:** NEPSE (Nepal Stock Exchange) publishes a "floorsheet" — a log of every trade that happened during the day. Traders need to see this data as live candlestick charts, volume analysis, and sector breakdowns in real-time (not after the market closes).

**Technical challenge:** You need to:
1. Ingest ~50,000 trades per trading day at high speed
2. Process them into 1-minute OHLCV candles in near real-time
3. Store millions of time-series data points efficiently
4. Query and visualize them with sub-second dashboard refresh

## 4. Every Component Explained Simply

### 4.1 Data Generator (`src/data_generator/`)

**What it does:** Creates realistic fake stock market data because we can't access real NEPSE data via API.

**How it works:**
- Uses **Geometric Brownian Motion (GBM)** for price simulation — the same math model used in financial engineering. Prices "random walk" up and down realistically.
- Generates a **U-shaped volume profile** — trading is heavy at market open (11 AM) and close (3 PM), light at midday. This matches real market behavior.
- Contains 65 real NEPSE stock symbols across 9 sectors, with realistic base prices and volumes.
- Output: A ~1GB CSV file with 12.5 million trade rows.

**Interview tip:** "I used Geometric Brownian Motion because it's the industry-standard model for simulating financial price movements. It produces realistic random walks with drift and volatility."

### 4.2 Kafka Producer (`src/producer/kafka_producer.py`)

**What it does:** Reads the CSV and sends each trade to Kafka at configurable speed (e.g., 100x = 4 hours of market data in 2.4 minutes).

**Key design choices:**
- **Chunked CSV reading** (10K rows at a time) — the file is 1GB, loading it all would use 4GB RAM
- **Symbol as message key** — this ensures all trades for one stock go to the same Kafka partition. Why? Because OHLCV needs trades in order (open = first price, close = last price). Kafka only guarantees ordering within a partition.
- **Snappy compression** — reduces network traffic by ~70%
- **Idempotent producer** — prevents duplicate messages if a retry happens
- **Delivery callbacks** — we track success/failure of every message

### 4.3 Kafka Broker (Docker)

**What it is:** Apache Kafka is a distributed message queue (like a post office for data).

**Why Kafka:**
- Handles millions of messages per second
- Stores messages durably (survives restarts)
- Decouples producers from consumers (they don't need to know about each other)
- Guarantees message ordering within a partition

**Why KRaft mode (no Zookeeper):**
- Kafka traditionally needed Zookeeper (a separate coordination service)
- KRaft (Kafka Raft) mode eliminates Zookeeper — one fewer container to manage
- It's the future direction of Kafka (Zookeeper is deprecated in Kafka 4.0)

**Topics in this project:**
- `nepse-raw-ticks` (3 partitions) — raw trade data
- `nepse-aggregated` (1 partition) — processed OHLCV data

### 4.4 PySpark Consumer (`src/consumer/spark_consumer.py`)

**What it does:** The brain of the pipeline. Reads raw trades from Kafka and computes 1-minute OHLCV candlestick aggregations.

**OHLCV explained:**
- **Open** = price of the FIRST trade in the 1-minute window
- **High** = HIGHEST price in the window
- **Low** = LOWEST price in the window
- **Close** = price of the LAST trade in the window
- **Volume** = total shares traded in the window

**Key concepts:**
- **Structured Streaming** — Spark treats the stream as a table that grows continuously. You write SQL-like operations on it.
- **Tumbling window** — Non-overlapping 1-minute windows. Trade at 11:00:30 goes into [11:00-11:01). Trade at 11:01:05 goes into [11:01-11:02).
- **Watermark** — "Wait up to 10 minutes for late-arriving data." If a trade arrives 5 minutes late (network delay), the window still accepts it. After 10 minutes, the window is finalized.
- **Output mode = "update"** — Only output windows that changed (more efficient than resending everything).

**Why PySpark (not Flink, ksqlDB, etc.):**
- DataFrame API makes aggregations clean and readable
- Event-time processing with watermarks handles late data
- Widely used in industry (valuable for resume)
- Available locally without cluster setup

### 4.5 TimescaleDB Sink (`src/sink/timescaledb_sink.py`)

**What it does:** Reads aggregated OHLCV data from Kafka and writes it to TimescaleDB.

**Why a separate Python script (not Spark's built-in JDBC sink):**
- Simpler: no JDBC driver JAR configuration (a major pain on Windows)
- Lighter: uses ~30MB RAM vs Spark's 500MB+ JVM overhead
- Idempotent upserts with `ON CONFLICT DO UPDATE` prevent duplicates
- Can restart independently of the Spark consumer

**Key pattern — batch upsert:**
- Accumulates 100 messages, then writes them all at once using `execute_batch()`
- This is ~10x faster than writing one by one (fewer database round-trips)
- `ON CONFLICT (symbol, window_start) DO UPDATE` — if the same candle arrives again (Spark sends updates), it replaces the old data instead of inserting a duplicate

**At-least-once delivery guarantee:**
- Kafka offsets are committed AFTER successful database write
- If the sink crashes before committing, it re-reads and reprocesses the messages
- Since upserts are idempotent, this is safe — no duplicates in the database

### 4.6 TimescaleDB (`sql/init.sql`)

**What it is:** PostgreSQL with time-series superpowers.

**Why TimescaleDB (not plain PostgreSQL):**
- **Hypertables** — automatically partitions data by time. When Grafana asks "show last 1 hour," it only scans the most recent chunk, not the entire table.
- **Continuous aggregates** — pre-computes 5-minute candles from 1-minute data. Grafana reads the pre-computed result instantly instead of re-calculating every 10 seconds.
- **Compression** — 90%+ storage reduction for historical data.
- Full PostgreSQL compatibility — works with Grafana, psycopg2, pgAdmin.

**Schema design:**
- `ohlcv_1min` table (hypertable partitioned by `window_start`)
- Unique constraint on `(symbol, window_start)` for idempotent upserts
- Indexes: `(symbol, time DESC)` and `(sector, time DESC)` for fast dashboard queries

### 4.7 Grafana Dashboard (`grafana/dashboards/`)

**What it is:** A live dashboard that auto-refreshes every 10 seconds.

**6 panels:**
1. **OHLC Candlestick** — price chart for selected stock (dropdown)
2. **Volume Bars** — shares traded per minute
3. **Pipeline Health** — total records, active symbols, last update
4. **VWAP by Sector** — donut chart showing average prices by sector
5. **Top 10 by Turnover** — most actively traded stocks
6. **Live Data Feed** — latest 50 OHLCV windows (streaming table)

**Auto-provisioning:** When you run `docker-compose up`, Grafana automatically:
1. Connects to TimescaleDB (via provisioned datasource YAML)
2. Loads the dashboard JSON (via provisioned dashboard config)
Zero manual clicking required.

## 6. Key Design Decisions

These are the things interviewers LOVE to ask about. Memorize the "why."

### "Why symbol as the Kafka message key?"
All trades for a stock (e.g., NABIL) go to the same Kafka partition. Kafka guarantees ordering within a partition. This is critical because OHLCV needs chronological order — "open" = first price, "close" = last price. If trades arrived out of order across partitions, our candles would be wrong.

### "Why 3 partitions?"
We use 3 Spark executor threads (`local[3]`). 1 partition = 1 Spark task. 3 partitions = 3 parallel consumers = maximum parallelism with zero contention. More partitions would create scheduling overhead with no benefit.

### "Why a decoupled Python sink instead of Spark JDBC?"
Three reasons: (1) No JDBC driver classpath issues (a major pain point on Windows), (2) The sink uses only ~30MB RAM vs Spark's 500MB+ JVM overhead, (3) It can restart independently without restarting the entire Spark pipeline.

### "Why ON CONFLICT DO UPDATE (idempotent upserts)?"
PySpark's `outputMode("update")` sends updated windows on every micro-batch. The same `(symbol, window_start)` pair may arrive multiple times as the window accumulates more trades. `ON CONFLICT` replaces the old partial window with the latest complete data, making the sink safe to re-process.

### "Why KRaft mode instead of Zookeeper?"
Kafka's KRaft mode eliminates the need for a separate Zookeeper service. Fewer containers = simpler operations. It's also the future direction — Zookeeper is deprecated in Kafka 4.0. Shows awareness of latest Kafka evolution.

### "Why TimescaleDB instead of plain PostgreSQL?"
Plain PostgreSQL scans ALL rows for time-range queries. TimescaleDB's hypertables auto-partition by time — so "show last 1 hour" only reads the latest chunk. With 65 symbols x 240 windows/day x 250 days = ~3.9M rows/year, this is essential for dashboard performance.

### "Why event-time processing with watermarks?"
In real market data feeds, trades can arrive out of order or with delays (network latency). Our 10-minute watermark says "wait up to 10 minutes for late data before finalizing a window." This ensures correctness even with disordered events — a production-critical requirement.

### "Why chunked CSV reading?"
The floorsheet CSV is 1+ GB. Loading it entirely into memory would use ~4GB RAM. By reading in chunks of 10,000 rows, we keep memory usage under 50MB. This is a standard pattern for processing large files.

### "Why Snappy compression on Kafka?"
Our JSON messages have repeated field names ("symbol", "rate", "amount") that compress extremely well. Snappy provides ~70% compression with minimal CPU overhead. This reduces network bandwidth and broker disk usage.

## 7. Data Model

### Raw Input (one trade)
```json
{
  "contract_no": "20210104-000001",
  "symbol": "NABIL",
  "buyer": 12,
  "seller": 35,
  "quantity": 100,
  "rate": 1150.50,
  "amount": 115050.00,
  "trade_time": "2021-01-04 11:00:00.004",
  "sector": "Commercial Bank"
}
```

### Processed Output (1-minute OHLCV candle)
```json
{
  "symbol": "NABIL",
  "window_start": "2021-01-04T11:00:00.000+05:45",
  "window_end": "2021-01-04T11:01:00.000+05:45",
  "open": 1150.50,
  "high": 1155.00,
  "low": 1148.00,
  "close": 1152.00,
  "volume": 5000,
  "turnover": 5762500.00,
  "trade_count": 45,
  "vwap": 1152.50,
  "sector": "Commercial Bank"
}
```

**VWAP (Volume-Weighted Average Price):** `total_turnover / total_volume`. Unlike a simple average, VWAP weighs prices by how many shares traded at each level. It's more accurate for financial analysis.

## 9. Common Interview Questions and Answers

### Architecture Questions

**Q: Walk me through the data flow.**
A: "Raw trades flow from CSV through a Kafka producer into the `nepse-raw-ticks` topic. PySpark Structured Streaming reads those trades, groups them into 1-minute tumbling windows per stock symbol, and computes OHLCV aggregations. The results are written to the `nepse-aggregated` topic. A standalone Python sink consumes from that topic and batch-upserts into TimescaleDB. Grafana queries TimescaleDB every 10 seconds to power the live dashboard."

**Q: Why did you use Kafka instead of just writing directly to the database?**
A: "Kafka decouples producers from consumers. If the database goes down, messages are safely stored in Kafka and can be processed later. It also allows multiple consumers to process the same data differently — one for aggregation, one for alerts, one for archival. Additionally, Kafka handles backpressure: if the consumer is slow, messages queue up instead of being lost."

**Q: How does your pipeline handle failures?**
A: "At each layer: (1) The producer has retries and idempotence — duplicates are prevented. (2) PySpark uses checkpointing — if it crashes, it resumes from where it left off. (3) The sink commits Kafka offsets only AFTER successful database write — if it crashes, it re-reads and re-processes. Since upserts are idempotent, this gives us at-least-once delivery with exactly-once effect at the storage layer."

**Q: What happens if a trade arrives late (out of order)?**
A: "PySpark's watermark handles this. I set a 10-minute watermark, meaning the system waits up to 10 minutes for late-arriving data before finalizing a window. A trade that arrives 5 minutes late is still included in the correct window. After 10 minutes, the window is closed and late data is dropped."

### Technical Deep-Dive Questions

**Q: Explain the difference between event time and processing time.**
A: "Event time is when the trade actually happened (e.g., 11:00:05 AM). Processing time is when Spark processes it (e.g., 11:00:15 AM). I use event time because it reflects reality — if Spark is slow or data arrives late, we still get correct windows based on when trades actually occurred."

**Q: What is a hypertable in TimescaleDB?**
A: "A hypertable is a virtual table that's automatically partitioned into chunks by time. From the user's perspective, it looks and behaves like a normal PostgreSQL table. But behind the scenes, data is split into 7-day chunks. When Grafana queries 'last 1 hour,' TimescaleDB only scans the most recent chunk instead of the entire table — this can be 100x faster."

**Q: What is a continuous aggregate?**
A: "It's a materialized view that TimescaleDB keeps automatically up-to-date. I have a 5-minute continuous aggregate that pre-computes 5-min candles from 1-min data. Instead of Grafana re-calculating this every 10 seconds, it reads the pre-computed result — instant response."

**Q: Why batch writes instead of writing one row at a time?**
A: "Each database write has overhead: network round-trip, transaction management, WAL logging. If I write 100 rows one-by-one, that's 100 round-trips. With `execute_batch()`, I send all 100 in a single round-trip — roughly 10x faster. At 19,504 total upserts with zero errors, the sink averages 183.6 rows/sec."

**Q: How would you scale this pipeline?**
A: "Horizontally at every layer: (1) Add more Kafka partitions and more Spark executor threads for higher parallelism. (2) Run multiple sink instances in the same consumer group — Kafka automatically balances partitions across them. (3) Add read replicas of TimescaleDB for dashboard queries. (4) In production, I'd containerize the Python components and use Kubernetes for orchestration."

### Behavioral / Design Questions

**Q: What would you do differently in production?**
A: See section 10 below.

**Q: What was the hardest challenge?**
A: "Getting PySpark to work on Windows. The Kafka connector JAR needs exact version matching between Spark (3.5.4) and Scala (2.12). Mismatched versions cause ClassNotFoundException at runtime. Also, Hadoop's NativeIO calls Windows DLLs that often fail. I solved this by bypassing NativeIO with JVM flags and implementing a local-first JAR resolution strategy that skips Maven network calls."

**Q: How do you ensure data quality?**
A: "Multiple layers: (1) The producer validates CSV data types via pandas dtype enforcement. (2) The Spark consumer's explicit schema rejects malformed JSON (null values instead of crashes). (3) The sink validates all 12 required fields before attempting a database write. (4) The database's UNIQUE constraint prevents duplicate candles. (5) The continuous aggregate's VWAP calculation uses NULLIF to avoid division-by-zero."

## 10. What I Would Improve in Production

1. **Schema Registry (Avro)** — Replace JSON with Avro for type safety and ~60% smaller messages
2. **Kubernetes** — Containerize all Python components and orchestrate with K8s
3. **Monitoring** — Add Prometheus metrics for producer throughput, consumer lag, sink error rate
4. **Alerting** — PagerDuty/Slack alerts when pipeline lag exceeds 5 minutes
5. **Exactly-once semantics** — Use Kafka transactions end-to-end
6. **Data retention** — Add TimescaleDB compression policy for data older than 30 days
7. **CI/CD** — GitHub Actions for automated testing and deployment
8. **Security** — Kafka SSL/SASL authentication, database credentials in a secret manager
9. **Backpressure handling** — Rate limiting the producer if Spark can't keep up
10. **Real data source** — Replace CSV replay with a live NEPSE API connector

## 11. Glossary of Terms

| Term | Simple Explanation |
|------|-------------------|
| **OHLCV** | Open, High, Low, Close, Volume — a candlestick summary of price action in a time window |
| **Kafka** | A distributed message queue — like a post office that never loses mail |
| **KRaft** | Kafka's new metadata management mode (replaces Zookeeper) |
| **Topic** | A named channel in Kafka where messages are published (like a mailbox label) |
| **Partition** | A subdivision of a topic. Messages in a partition are strictly ordered |
| **Consumer Group** | Multiple consumers sharing the work of reading a topic (each gets some partitions) |
| **PySpark** | Python API for Apache Spark — a distributed computing engine |
| **Structured Streaming** | Spark's stream processing engine — treats a stream like a growing table |
| **Tumbling Window** | A non-overlapping time window (e.g., [11:00-11:01), [11:01-11:02)) |
| **Watermark** | How long to wait for late data before finalizing a window |
| **Hypertable** | TimescaleDB concept — a table auto-partitioned by time for fast queries |
| **Continuous Aggregate** | A pre-computed view that auto-refreshes as new data arrives |
| **Upsert** | INSERT or UPDATE — insert if new, update if already exists (`ON CONFLICT`) |
| **Idempotent** | Can be run multiple times with the same result (safe to retry) |
| **VWAP** | Volume-Weighted Average Price — average price weighted by shares traded |
| **GBM** | Geometric Brownian Motion — mathematical model for simulating stock prices |
| **Floorsheet** | NEPSE's public log of every trade executed on the exchange |
